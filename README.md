# Emperor

> **Work in Progress (Pre-release)** — This project is under active development and has not reached v1.0. Features, APIs, and configurations may change.

A PyTorch-based deep learning framework for experimenting with neural network architectures including Mixture of Experts (MoE), Vision Transformers (ViT), adaptive computation, and linear classifiers. Built on PyTorch Lightning for streamlined training across multiple image datasets.

Built as a personal research framework for exploring how architectural choices affect model behaviour, and for learning how to design extensible deep learning software.

## Quick Start

```bash
# 1. Set up the environment
source env.sh

# 2. Run a simple experiment
source run_experiment.sh linear --name DEFAULT

# 3. View training metrics
tensorboard --logdir logs/linear
```

## The Task

Emperor provides a modular system for building and evaluating neural network architectures on image classification tasks. Each experiment trains a model across standard benchmark datasets (MNIST, FashionMNIST, CIFAR-10, CIFAR-100) and logs metrics to TensorBoard.

The framework supports the following model architectures:

- **linear** - Fully-connected classifier with configurable depth, activation, and dropout
- **linear_adaptive** - Fully-connected classifier where a generator network produces auxiliary parameters (diagonal weights, bias, memory) that are combined with the default layer parameters at each layer
- **experts** - Mixture of Experts architecture with learned routing between specialized sub-networks
- **vit** - Vision Transformer that splits images into patches and applies transformer encoder blocks
- **parametric_generator** - Adaptive parameter layer where expert weights are generated by a generator network
- **parametric_matrix** - Adaptive parameter layer where expert weights are mixed as full matrices via a shared router
- **parametric_vector** - Adaptive parameter layer where expert weights are mixed as vectors via independent routers with configurable layer stack options
- **bert** - BERT-style language model with transformer encoder, trained on text datasets (PennTreebank, WikiText2)

Each architecture defines preset configurations and a search space for hyperparameter exploration via grid search or random sampling.

## Environment Setup

**Requirements:** 3.13

**Note:** Windows users must use [WSL](https://learn.microsoft.com/en-us/windows/wsl/install) to run the setup scripts.

### Using env.sh (recommended)

```bash
source env.sh
```

This will:

1. Install `mise` if not already installed (for Python version management)
2. Install Python 3.13 via `mise`
3. Create a virtual environment at `./torchenv`
4. Install pip and all project dependencies from `pyproject.toml`
5. Activate the virtual environment

On subsequent runs, it simply activates the existing environment.

### Manual setup

```bash
curl https://mise.run | sh
export PATH="$HOME/.local/bin:$PATH"
mise install
python3 -m venv torchenv
source torchenv/bin/activate
pip install --upgrade pip
pip install -e .
```

## Running Experiments

Use `run_experiment.sh` to train models. Each experiment runs across all four image datasets by default (MNIST, FashionMNIST, CIFAR-10, CIFAR-100). To change which datasets are used, edit the `DATASET_OPTIONS` list in the model's `config.py`:

```python
# models/linear_adaptive/config.py
DATASET_OPTIONS: list = [Mnist, FashionMNIST, Cifar10, Cifar100]
```

### List available experiments

```bash
source run_experiment.sh --list
```

```
Available experiments:
  bert
  experts
  linear
  linear_adaptive
  parametric_generator
  parametric_matrix
  parametric_vector
  vit
```

### Run a configuration

```bash
source run_experiment.sh linear --name DEFAULT
source run_experiment.sh vit --name ADAPTIVE
```

`--name` selects which experiment option to run. Each model exposes its own set of named options. Use `--name` or `--all-options` — one is required.

Available options per model:

| Model                  | Options                                                                         |
| ---------------------- | ------------------------------------------------------------------------------- |
| `linear`               | `PRESET`, `CONFIG`                                                              |
| `linear_adaptive`      | `PRESET`, `CONFIG`, `GENERATOR_DEPTH`, `DIAGONAL`, `BIAS`, `MEMORY`, `COMBINED`, `WEIGHT`, `WEIGHT_NORMALIZATION`, `ROW_MASK` |
| `experts`              | `PRESET`, `CONFIG`, `ADAPTIVE`                                                  |
| `vit`                  | `PRESET`, `CONFIG`, `ADAPTIVE`                                                  |
| `bert`                 | `PRESET`, `CONFIG`                                                              |
| `parametric_generator` | `PRESET`, `CONFIG`                                                              |
| `parametric_matrix`    | `PRESET`, `CONFIG`                                                              |
| `parametric_vector`    | `PRESET`, `CONFIG`                                                              |

For `linear_adaptive`, each option is a focused ablation experiment:

- `PRESET` — runs a single configuration using the default values defined in `config.py`
- `CONFIG` — searches over general hyperparameters (learning rate, hidden dim, activation, etc.)

### Run all configurations

```bash
source run_experiment.sh linear_adaptive --all-options
```

Runs every option in `ExperimentOptions` sequentially across all datasets.

### Run grid search

```bash
source run_experiment.sh linear --name BASE --grid-search
source run_experiment.sh linear_adaptive --all-options --grid-search
```

Exhaustively runs every combination in the search space across all datasets.

The search space is defined in each model's `config.py` as variables prefixed with `SEARCH_SPACE_`:

```python
# models/linear/config.py
SEARCH_SPACE_LEARNING_RATE = [1e-4, 1e-3, 1e-2]
SEARCH_SPACE_HIDDEN_DIM = [64, 128, 256]
SEARCH_SPACE_STACK_ACTIVATION = [RELU, SILU, GELU, MISH]
...
```

### Run random search

```bash
source run_experiment.sh linear --name BASE --random-search 10
source run_experiment.sh linear_adaptive --all-options --random-search 10
```

Randomly samples 10 configurations from the search space. `--grid-search` and `--random-search` are mutually exclusive and can be combined with either `--name` or `--all-options`.

### Custom log folder

```bash
source run_experiment.sh linear --log-folder comparison_run
source run_experiment.sh vit --log-folder comparison_run
```

Use the same `--log-folder` across different models to group their TensorBoard logs together for comparison.

### Viewing results

Training metrics are logged to `logs/`. Launch TensorBoard to visualize them:

```bash
tensorboard --logdir logs/
```

### Search results

When running `--grid-search` or `--random-search`, two additional files are written:

- **`result.json`** — saved inside each run's log directory with the exact parameters and final metrics for that run
- **`best_results.json`** — saved at `logs/{model}/best_results.json`, updated after every run that makes the top 5

`best_results.json` tracks the top 5 configurations per dataset ranked by `val_accuracy`:

```json
{
  "Mnist": [
    {"rank": 1, "option": "CONFIG", "params": {"learning_rate": "0.001", ...}, "metrics": {"val_accuracy": 0.984, "val_loss": 0.051, ...}},
    {"rank": 2, ...},
    ...
  ],
  "Cifar10": [...]
}
```

The file is updated in place as the search runs — if the process is interrupted, the top 5 found so far is preserved.

### Feature Roadmap

- **Wrappers**
  - [x] `Layer` — wraps a layer with configurable normalization, activation, dropout, and residual connections
    - [x] `Activation` — configurable activation function (ReLU, SiLU, GELU, Mish, etc.)
    - [x] `LayerNorm` — layer normalization applied before, after, or at the default position relative to the transform
    - [x] `Dropout` — randomly zeroes elements during training to reduce overfitting
    - [x] `ResidualConnection` — adds the layer input directly to its output, enabling gradient flow across depth
    - [ ] Dynamic gate option
  - [x] `LayerStack` — sequential pipeline of initial, hidden, and output layers with configurable depth and dimensions
- **Linear transformation**
  - [x] `LinearLayer` — standard affine transform (y = Wx + b) with optional bias
  - [x] `AdaptiveLinearLayer` — affine transform where weights and bias are dynamically modified per sample via augmentation modules
    - [x] `DynamicDepth` — controls how many generator-produced parameter adjustments are combined with the base weights
    - [ ] Single or dual parameter generators
    - [ ] Default weight parameters decay option
    - [ ] Input dependent depth options
    - [x] `DynamicDiagonal` — adds input-dependent adjustments along diagonal positions of the weight matrix
      - [x] `DIAGONAL` — adjustments placed along the main diagonal of the weight matrix
      - [x] `ANTI_DIAGONAL` — adjustments placed along the anti-diagonal (flipped) of the weight matrix
      - [x] `DIAGONAL_AND_ANTI_DIAGONAL` — applies both main and anti-diagonal adjustments sequentially
    - [x] `DynamicBias` — generates or modifies the bias vector per sample based on the input
      - [x] `SCALE_AND_OFFSET` — generates scale and offset values to affine-transform the existing bias (scale * bias + offset)
      - [x] `ELEMENT_WISE_OFFSET` — generates a per-element offset vector added directly to the existing bias
      - [x] `DYNAMIC_PARAMETERS` — generates the entire bias vector from scratch via a generator network
      - [x] `GATED` — applies a learned sigmoid gate to selectively scale each element of the existing bias
      - [ ] Bias weight bank with router option
    - [x] `DynamicWeight` — generates per-sample weight matrix adjustments via configurable decomposition strategies
      - [x] `DUAL_MODEL` — two generators produce input and output vectors whose outer product adjusts the weight matrix
      - [x] `SINGLE_MODEL` — one generator produces a vector whose self outer product forms a symmetric weight adjustment
      - [x] `LOW_RANK` — two generators produce low-rank factors multiplied together for a rank-decomposed weight update
      - [x] `WEIGHT_MASK` — two generators produce vectors whose outer product creates an element-wise multiplicative mask over the weights
      - [x] `HYPERNETWORK` — a single generator network produces the full weight matrix adjustment in one pass
    - [x] `WeightNormalization` — constrains augmented weight magnitudes using configurable normalization to stabilize training
      - [x] `CLAMP` — hard-clips weight values to a fixed symmetric range
      - [x] `L2_SCALE` — normalizes weights by their L2 norm then re-scales via a learnable parameter
      - [x] `SOFT_CLAMP` — smoothly bounds weight values using tanh for differentiable range limiting
      - [x] `RMS` — normalizes weights by dividing by their root mean square value
      - [x] `SIGMOID_SCALE` — maps weight values through sigmoid to bound them within (0, 1)
    - [x] `RowMask` — dynamically masks weight matrix rows to control which output dimensions are active per sample
      - [x] `GLOBAL_SCORE` — computes a global score vector and selects top-k rows by magnitude
      - [x] `PER_ROW_SCORE` — each row receives an independent sigmoid score for binary activation
      - [x] `TOP_SLICE` — retains the first N rows by index position, zeroing the rest
    - [x] `Memory` — enriches the input by blending it with a trainable memory representation
      - [x] `FUSION` — concatenates input with learned memory and compresses back to original dimension via an MLP
      - [x] `WEIGHTED` — blends input and memory features using softmax-computed attention weights
      - [x] `MemorySize` — controls the dimensionality of the learned memory vector (4, 8, 16, or 32)
      - [x] `MemoryPosition` — determines whether memory is blended with the input before or after the linear transform
    - [ ] Dynamic parameters batch sub-set
- **Input-dependent parameter generation**
  - [x] `AdaptiveParameterLayer` — generates per-sample weights and biases through routed mixture models with optional augmentation
    - [x] `AdaptiveWeight` — produces per-sample weight matrices by routing to and mixing from a bank of weight experts
      - [x] `VECTOR` — independent routers per component select and combine weight vectors from separate expert banks
      - [x] `MATRIX` — a shared router selects and blends full weight matrices from an expert parameter bank
      - [x] `GENERATOR` — a generator network produces weight matrices directly from the input, with no stored expert bank
    - [x] `AdaptiveBias` — produces per-sample bias vectors by routing to and mixing from a bank of bias experts
      - [x] `MATRIX` — a shared router selects and blends bias vectors from an expert bias bank
      - [x] `GENERATOR` — a generator network produces bias vectors directly from the input, with no stored expert bank
- **Sampling and routing**
  - [x] `RouterModel` — projects input tokens to per-expert logit scores via a learned MLP for downstream sampling
    - [x] `NoisyTopK` — injects tunable Gaussian noise into router logits before selection to encourage expert exploration
  - [x] `SamplerModel` — converts router logit scores into expert selection indices, probabilities, and auxiliary losses
    - [x] `SamplerSparse` — routes each token to exactly one expert via top-1 selection
    - [x] `SamplerTopk` — routes each token to the k highest-scoring experts and normalizes their probabilities
    - [x] `SamplerFull` — sends every token to all experts, weighting contributions by softmax probabilities
    - [x] `ThresholdMasking` — zeros out expert-token assignments whose selection probability falls below a configurable threshold
    - [x] `RandomSampling` — randomly substitutes a portion of top-k expert selections to promote routing diversity
  - [x] `AuxiliaryLosses` — regularization losses that encourage balanced and diverse expert utilization
    - [x] `CoefficientOfVariation` — penalizes high variance in per-expert token counts to encourage uniform load distribution
    - [x] `SwitchLoss` — penalizes imbalanced routing by combining per-expert dispatch fractions with router probabilities
    - [x] `ZeroCentred` — regularizes router logits toward zero mean to prevent score drift and saturation
    - [x] `MutualInformation` — maximizes mutual information between inputs and expert assignments to diversify routing patterns
- **Mixture of Experts**
  - [x] `MixtureOfExperts` — dispatches tokens to top-k selected experts, processes them independently, and aggregates weighted outputs
    - [x] `ExpertWeighting` — scales expert contributions by their router-assigned selection probabilities
      - [x] `BEFORE_EXPERTS` — scales token representations by selection probabilities before expert processing
      - [x] `AFTER_EXPERTS` — scales expert outputs by selection probabilities after expert processing
    - [x] `ExpertCapacity` — caps the number of tokens each expert processes per batch, dropping or preserving overflow
      - [x] `ZEROS` — replaces overflow tokens that exceed expert capacity with zero vectors
      - [x] `IDENTITY` — overflow tokens bypass expert processing and retain their original input representations
    - [x] `InitSampler` — controls whether router and sampler instances are shared across MoE layers or created per layer
      - [x] `SHARED` — all MoE layers in the stack share a single router and sampler instance
      - [x] `LAYER` — each MoE layer creates its own router and sampler for layer-specific routing decisions
  - [x] `MixtureOfAttentionHeads` — applies expert routing to attention heads, selecting a subset of heads per token
    - [x] `MixtureOfExpertsMap` — routes tokens to top-k attention head experts and returns individual expert outputs without combining
    - [x] `MixtureOfExpertsReduce` — aggregates top-k expert outputs back to the original shape via probability-weighted summation
  - [ ] Soft mixture of experts from [From sparse to soft mixtures of experts](https://arxiv.org/pdf/2308.00951)
- **Attention**
  - [x] `MultiHeadAttention` — computes scaled dot-product attention across multiple heads with configurable Q/K/V projection strategies
    - [x] `SELF_ATTENTION` — generates Q, K, and V from a single shared projection of the input
    - [x] `INDEPENDENT` — uses separate projection matrices for Q, K, and V, supporting cross-attention
    - [x] `MIXTURE_OF_ATTENTION_HEADS` — selects a subset of attention heads per token via mixture-of-experts routing
  - [ ] Implement [MLP-Mixer - An all-MLP Architecture for Vision](https://arxiv.org/pdf/2105.01601) but with `AdaptiveLinearLayer`
  - [ ] Implement [Atlas: Learning to Optimally Memorize the Context at Test Time](https://arxiv.org/pdf/2505.23735)
  - [ ] Add experience matrix-based similar to rnn hidden state
- **Transformer**
  - [x] `PatchEmbedding` — divides images into fixed-size patches and projects each into a token embedding
    - [x] `LINEAR` — unfolds image patches into flat vectors and maps them to embeddings via a linear layer
    - [x] `CONV` — extracts and embeds patches using a convolution with kernel and stride matching patch size
  - [x] `FeedForward` — transformer feed-forward sub-layer wrapping a configurable layer stack
    - [x] `Linear` — feed-forward block built from standard linear layers
    - [x] `Adaptive` — feed-forward block built from adaptive linear layers with per-sample parameter modification
    - [x] `MixtureOfExperts` — feed-forward block built from mixture-of-experts routed sub-networks
  - [x] `TransformerEncoderLayer` — single encoder block combining self-attention over source tokens with a feed-forward network
  - [x] `TransformerDecoderLayer` — single decoder block with target self-attention, encoder cross-attention, and feed-forward
  - [x] `TransformerEncoderStack` — stacks multiple encoder layers sequentially to build depth in the encoder
  - [x] `TransformerDecoderStack` — stacks multiple decoder layers sequentially to build depth in the decoder
  - [x] `Transformer` — complete encoder-decoder architecture with embedding, positional encoding, and output projection
  - [ ] Latent space reasoning similar to [Less is More: Recursive Reasoning with Tiny Networks](https://arxiv.org/pdf/2510.04871)
- **Positional embeddings**
  - [x] `AbsolutePositionalEmbedding` — adds position-dependent vectors to token embeddings to encode sequential or spatial order
    - [x] `SINUSOIDAL` — fixed position encoding using interleaved sine and cosine functions at geometrically-spaced frequencies
    - [x] `LEARNED` — position embeddings stored as trainable parameters optimized during training
  - [x] `RelativePositionalEmbedding` — adds distance-dependent bias terms to attention scores to encode relative token positions
    - [x] `DynamicPositionalBias` — generates per-head relative position biases via a learned MLP for flexible distance encoding
- **Halting**
  - [x] `SoftHalting` — accumulates per-token halting probabilities across layers with an ACT loss penalizing excess computation depth
  - [x] `StickBreaking` — tokens stop updating once accumulated halt probability exceeds a threshold, using hard masks to freeze state
- **Neuron**
  - [x] `Nucleus` — central computation unit of a neuron that applies a linear transform to the input signal
  - [x] `Axons` — output transformation stage applied to the nucleus signal (extensible, currently pass-through)
  - [x] `Terminal` — routes signals to neighboring neurons in a 3D spatial grid via learned router and sampler
  - [x] `Neuron` — biologically-inspired unit composing nucleus (compute), axons (output), and terminal (routing)
  - [x] `NeuronCluster` — 3D spatial grid of interconnected neurons with dynamic growth of new neurons at empty neighbor positions
  - [ ] Wrap `Neuron` or into `halting` mechanism
  - [ ] Grow new neurons based on neighboring neurons' training maturity
  - [ ] Freeze lower neurons and only train the top of the stack
  - [ ] Sparse-train neurons that have been disconnected from the gradient
    - [ ] Select the most salient representation from the batch
    - [ ] Write the selected representation to a memory matrix via single-head attention
    - [ ] Flush the memory matrix to the neuron once a gating model determines sufficient information has accumulated
